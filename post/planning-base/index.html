Planning for an objective: Base system

Tue, Jan 11, 2022

If there’s something remotely important, it helps to have a plan1. This is the system I use for planning.

Assumptions

There are scenarios where this system doesn’t work perfectly, but with minor modifications the system can respond to the new scenarios. I’ll go into how to handle those scenarios in another post, but before we augment the system we have to lay out the system. In order to layout the system here are some simplifying assumptions.

  • The planner knows the domain
  • The planner knows the existing system
  • The planner knows the objective isn’t going to change

The plan for planning

Roughly block out the piece work → Break down each piece of work→ Recurse until each unit of work is no more than one or two days.

Blocking

Using the objectives, domain, and understanding of the current system, figure out each major section of work. In addition, work should be broken up so that each section is testable. This is probably the hardest part because sometimes something complex has many moving parts. In those cases I recommend getting a skeleton setup and having a sample output be the verification (e.g. returns a default message or logs a message).

I’ve found these blocks of work fall into roughly the same categories. Below are several that I’ve found useful.

Functional blocks

These are the pieces of work about getting the actual thing done. For example, it could be building out a specific page, adding a new feature, or modifying the product. I generally think of these as the pieces of work you can point to in the software.

Technical blocks

These are the pieces of work behind the scenes that are necessary to get the functional blocks working. It could be setting up a framework or library, it could be refactoring the old code, it could be setting up a new integration required for functionality.

Process blocks

These are the pieces of work about the work. For example, it could be the work required for releasing the software2.

Arranging the blocks

Each block is a conceptual unit of work. Almost every time, blocks are going to be dependent on other blocks, forming chains of work. Once you have the blocks, figure out which blocks need to be completed first. For example, you probably need to set up your API server framework before you can build out integration with Stripe.

Tip: Draw the blocks. Use arrows to figure out the dependencies between them. This will let you know what can be parallelized and how to start.

Note: The block diagram should be DAG. If it’s not then make it one. We have to start somewhere.

Breaking down

Each block has its own large piece of work. Within the block are a bunch of smaller things that need to happen. For example, if you’re setting up an api framework you might need to set up the logging middleware or add authentication. Each major section is it’s own unit of work, albeit smaller. Once all the smaller pieces are broken down then again figure out the dependencies.

Recursing

I recommend stopping planning once each unit of work is about 1 or 2 days effort. If the broken down blocks are too big then do the process again. If the broken down blocks are too small then add the details as helpful notes.

  1. Just because there is a plan doesn’t mean I stick to the plan. Getting to the objective is more important than sticking to the plan. Having a plan dramatically increases the chances of getting the objective done. ↩︎

  2. Some radical engineers may be saying everything should be instantly deployed once it’s merged. That’s nice if you work on some non-critical website selling junk we don’t need, but some folks work on code that falls under FDA regulation, in which case there is non-trivial process work to do. ↩︎
